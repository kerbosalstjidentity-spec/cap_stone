# FDS 파이프라인: 현재 범위 vs 실제 운영 시스템

## 개요

이 문서는 캡스톤 프로젝트에서 구현한 FDS(Fraud Detection System) 파이프라인과 실제 신용카드 결제 시스템의 사기 탐지 흐름을 비교하며, 각 갭이 기술적 선택인지 인프라 제약인지 명확히 한다.

---

## 현재 구현: 싱글 트랜잭션 기반 파이프라인

```
[거래 1건]
    ↓
[모델 스코어 (Random Forest)]
    ↓
[규칙 엔진 (3가지: 금액, PSI 드리프트, 정책)]
    ↓
[액션 결정: PASS / REVIEW / BLOCK]
```

### 특징
- **입력**: 거래 1건의 특성만 (V1–V30 또는 V1–V339)
- **모델**: 단일 RF 번들 (SMOTE 적용, Stratified 5-fold CV 검증)
- **규칙**: 3가지 정책 (임계값 + PSI 드리프트 + 구조적 검증)
- **출력**: CSV 형식의 배치 스코어 & 액션 결정
- **피드백**: 라벨 파일이 주어질 때만 사후 성능 측정 (`ops_labeled_metrics.py`)

### 구현 수준의 신뢰도
| 항목 | 상태 | 근거 |
|------|------|------|
| 모델 선택 신뢰도 | ✅ 높음 | 5-fold CV, PR-AUC 기반 평가 |
| 정책 수정 용이성 | ✅ 높음 | YAML 기반, 코드 변경 불필요 |
| 데이터 누수 방지 | ✅ 높음 | SMOTE는 fold 내에만 적용, 검증은 원본 분포 |
| 드리프트 감지 | ✅ 높음 | PSI 5등급 기준, 자동 경보 |

## 실제 운영 시스템: 컨텍스트 기반 앙상블 파이프라인

```
[거래 1건]
    + [이용자 최근 30일 패턴] (velocity, 금액 range, 가맹점 선호도)
    + [기기 기반 그래프] (같은 기기 → 다른 계좌들)
    + [가맹점 리스크 점수] (해당 가맹점 사기율, 산업군)
    + [시공간 정보] (거래 시간대, 결제 위치 vs 주소지 거리)
    ↓
[모델 앙상블 N개] (RF, LGBM, NN, 룰 기반 점수 등)
    ↓
[규칙 수백 개] (velocity 체크, 카드 유무 거래, 반복 거래, 네트워크 분석 등)
    ↓
[심사원 검토 큐] (high-touch 케이스 → 콜센터 확인)
    ↓
[결제 승인/거부 연동] (실시간 API ← → 카드사/가맹점)
    ↓
[피드백 루프] (결과 라벨 → 매일 재학습)
```

### 구성 요소 상세

| 계층 | 항목 | 실제 필요 리소스 |
|------|------|-----------------|
| **맥락 데이터** | 고객 프로필 (30일 거래 패턴) | Historical DB + 스트림 집계 |
| | 기기 지문 (DeviceID ↔ 계좌 매핑) | 그래프 DB (Neo4j 등) |
| | 가맹점 리스크 점수 | Master DB + 외부 피드 |
| | 지리정보 (거래지 ↔ 주소 거리) | Geocoding API |
| **모델 계층** | 앙상블 (RF + LGBM + NN) | ML Registry, A/B 테스트 인프라 |
| | 실시간 추론 | 초저지연 serving (ms 단위) |
| **규칙 엔진** | 동적 규칙 로딩 | 규칙 관리 시스템 (RMS) |
| | 온라인 A/B 테스트 | 실험 플랫폼 |
| **인간 루프** | 심사원 UI + 라우팅 | 콜센터 통합, SLA 관리 |
| **피드백** | 일일/주간 재학습 | Airflow + cron 배치 |
| | 라벨 품질 관리 | 분쟁 해결 시스템 |

## 갭 분석: 왜 이 차이가 생기는가

### 1. 컨텍스트 피처 부재 (인프라 제약 ⚠️)

**현재 상황:**
```python
# 입력: 단일 거래 데이터만
transaction = {"V1": 0.5, "V2": -0.3, ..., "V30": 0.1}
score = model.predict_proba(transaction)[0, 1]
```

**실제 필요:**
- 고객 최근 30일 거래 패턴
- 같은 기기로 접속한 다른 계좌 목록
- 가맹점별 사기율
- 현재 위치 vs 청구 주소 거리

**왜 안 했나:**
- `data/open/val.csv`와 `data/ieee-cis/`는 **시계열·그래프·마스터 데이터 없음**
- 이들을 추가하려면 실제 은행 데이터 필요 (민감 정보 문제)
- 또는 시뮬레이션용 데이터셋 자체를 구성해야 함

**코드로 시뮬레이션 가능한 수준:**
- `creditcard.csv`의 `Time` 컬럼으로 **velocity** 역산 (1시간 거래 횟수 등)
- 그것도 시간 순서 보장 필요 → 현재 데이터는 이미 섞여 있음

**향후 추가 방안:** `docs/FDS_V2_MOCK.md` 참고 (모의 30일 패턴 데이터 생성)

---

### 2. 모델 앙상블 부재 (코드 추가 가능 ✅)

**현재 상황:**
```python
# 단일 모델: Random Forest만
model = RandomForestClassifier(...)
score = model.predict_proba(X)[0, 1]
```

**실제 필요:**
- RF, LGBM, Neural Net 등 여러 모델
- 앙상블 가중치 (평균 vs stacking 등)
- 모델별 성능 차이 반영

**왜 안 했나:**
- 단일 모델(RF)로도 충분한 성능 확보 (PR-AUC 0.79)
- 앙상블 추가는 "수치 +2%"일 가능성 높음 (ROI 낮음)
- 오버피팅 위험 증가

**추가 난이도:** 낮음 (LGBM 학습·앙상블 평균 추가)

**향후 추가 방안:** `train_export_rf.py`를 `train_export_ensemble.py`로 확장

---

### 3. 규칙 수 제한 (코드 추가 가능 ✅)

**현재:**
- 금액 임계값 (액수 기반)
- PSI 드리프트 (피처 분포 변화)
- 구조적 검증 (필수 필드 등)
→ **총 3가지 정책**

**실제 시스템:**
- Velocity 규칙 (1시간 거래 횟수, 일일 금액 누적)
- 카드 유무 거래 (온라인 vs 오프라인 패턴)
- 반복 가맹점 (평소 가는 곳이 아닌가)
- 네트워크 규칙 (같은 기기 다른 계좌와 유사 패턴)
- 카드 상태 규칙 (최근 신청·한도 변경 등)
→ **수백 개**

**왜 안 했나:**
- velocity 규칙은 시간 순서 필요 (현재 데이터 크로스섹션)
- 가맹점 마스터 데이터 필요
- 각 규칙마다 threshold 튜닝 필요

**코드로 시뮬레이션 가능한 수준:**
- `creditcard.csv`를 시간순으로 정렬해 velocity feature 생성
- rule_engine.py에 velocity 규칙 몇 개 추가

**향후 추가 방안:** `rules_engine.py`에 velocity·가맹점 카테고리 기반 규칙 추가

---

### 4. 심사원 검토 루프 부재 (인프라 + 인사 제약 ⚠️)

**현재:**
```
REVIEW 액션 결정 → CSV로 큐 추출 (export_review_queue.py)
→ 사람이 CSV 열어서 봄 (또는 바이너리 API로 응답)
```

**실제:**
```
REVIEW 결정 → 심사원 UI 라우팅
→ 콜센터 직원 확인 (3분 내)
→ 고객 확인 전화 또는 SMS OTP
→ 승인/거부 결정 → 카드사 API 호출
→ 결과 라벨화 (수락/거부) → 재학습 데이터
```

**왜 안 했나:**
- 콜센터 시스템 필요 (실제 인프라)
- 심사원 역할 수행 불가 (인사 제약)
- 실시간 고객 응답 필요 (시뮬레이션 불가)

**코드로 시뮬레이션 가능:**
- 불가능 (인간의 역할이므로)
- 모의 심사원 결정 생성 가능 (but 의미 제한적)

---

### 5. 결제 시스템 연동 부재 (외부 시스템 제약 ⚠️)

**현재:**
```python
# 파이프라인 독립 실행 (배치)
decisions = batch_decide(scores, thresholds)
decisions.to_csv("decisions.csv")
```

**실제:**
```
BLOCK → 거래 즉시 거부 (카드사 승인 거절)
PASS → 거래 승인
REVIEW → 1초 내 결과 반환 (콜센터 확인 대기)
```

**왜 안 했나:**
- 결제 게이트웨이 API 필요
- 실제 카드 결제 불가 (PCI-DSS 규제)
- 트랜잭션 타이밍 문제 (ms 단위 제약)

**코드로 시뮬레이션 가능:**
- API 스텁 생성 가능 (FastAPI 같은 것으로)
- 하지만 실제 효과 없음

---

### 6. 지속적 재학습 루프 부재 (인프라 제약 ⚠️)

**현재:**
```
1회 학습 (train_export_rf.py)
→ 모델 번들 저장
→ 배치 추론 (batch_run.py)
→ (선택) 라벨 수집 후 사후 평가
```

**실제:**
```
Day 1: 초기 모델 배포
Day 2–365:
  - 어제 거래 라벨 수집
  - 하루마다 재학습 (Airflow 스케줄)
  - 모델 성능 모니터 (Grafana)
  - 드리프트 감지 시 경보
  - A/B 테스트로 신 모델 배포 (10% 트래픽)
```

**왜 안 했나:**
- 일일 배치 스케줄러 필요 (Airflow, Kubeflow)
- MLOps 기반구조 필요 (모델 저장소, 메타데이터 추적)
- 지속적 라벨 수집 파이프라인 필요

**코드로 시뮬레이션 가능:**
- cron 작업 또는 간단한 스케줄러로 "모의 일일 재학습" 구현
- 하지만 실제 라벨 피드백 없음 (모의 데이터 사용)

## 정리: 갭의 분류

| 갭 | 원인 | 영향 | 향후 추가 난이도 |
|-----|------|------|-----------------|
| **컨텍스트 피처** | 데이터 제약 | 낮음 (성능 하지만 기준 충족) | 높음 (데이터 수집/시뮬레이션) |
| **모델 앙상블** | 설계 선택 | 극히 낮음 | 낮음 (코드 추가) |
| **규칙 수백 개** | 설계 선택 | 낮음 (3개로도 정책 충분) | 중간 (규칙 로직 추가) |
| **심사원 루프** | 인프라/인사 | 높음 (실제 시스템 중요) | 높음 (실제 인력 필요) |
| **결제 연동** | 외부 시스템 | 높음 (결정 실행 불가) | 높음 (API 스텁 가능, but 효과 제한) |
| **재학습 루프** | 인프라 | 높음 (장기 성능) | 중간 (스케줄러 추가) |

## 캡스톤 프로젝트로서의 의의

### ✅ 올바르게 구현된 것

1. **모델 신뢰도**: Stratified 5-fold CV + PR-AUC 기반 평가
2. **정책 분리**: 점수 & 규칙을 YAML로 독립 관리
3. **드리프트 감지**: PSI 자동 경보 시스템
4. **테스트 커버리지**: E2E 22건, PSI 수학 6건, 라벨 메트릭 8건

### ⚠️ 인식하고 있지만 스코프 밖

1. 컨텍스트 피처 (데이터 제약)
2. 심사원 루프 (인사 제약)
3. 실시간 결제 연동 (외부 시스템)
4. 일일 재학습 자동화 (인프라)

### 🔄 추가 가능하면 좋을 것 (낮은 난이도)

1. LGBM 앙상블 (모델 평가 확대)
2. Velocity 규칙 몇 개 (시간 기반 feature)
3. 간단한 스케줄 재학습 시뮬레이션
