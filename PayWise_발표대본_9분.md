## 🎯 슬라이드 3 — 프로젝트 개요 `[0:40 ~ 2:00]` ⏱ **1분 20초**

먼저 문제 정의입니다.

현재 대부분의 카드 앱은 **내역 조회에만 머무르고 있습니다.** 소비를 분석해 주지 않고, 이상 거래가 발생해도 사후 알림 수준이라 금전적 피해를 막기 어렵습니다. 개인화된 예산 조언도 없고, **소비 패턴과 사기 탐지가 연동된 서비스는 사실상 전무한 상황**입니다.

저희는 이 네 가지 문제를 AI로 풀고자 했습니다.

중앙 솔루션 박스를 보시면,
- **K-Means**로 사용자의 소비 유형을 절약형·균형형·소비형·투자형 네 그룹으로 분류하고,
- **Isolation Forest**로 이상 거래를 실시간 탐지하며,
- **LSTM과 XGBoost**로 지출을 예측·분류합니다.
- 여기에 **규칙 엔진과 ML을 결합한 하이브리드 사기 탐지** 모델을 구축했습니다.

하단의 네 가지 핵심 목표 — AI 분석, 실시간 탐지, 개인화 전략, 유형 군집화 — 이것이 PayWise가 해결하려는 문제입니다.

## 🎯 슬라이드 4 — 시스템 아키텍처 `[2:00 ~ 3:20]` ⏱ **1분 20초**

시스템 전체 구조입니다. 크게 **프론트엔드 · 백엔드 · ML 엔진 · 연동 서비스** 네 영역으로 구성했습니다.

프론트엔드는 **Next.js 14 App Router** 기반이며 TypeScript와 Recharts로 차트를 렌더링합니다. 백엔드는 **FastAPI**를 택했는데, 비동기 I/O와 Swagger 자동 문서화가 ML 서빙에 최적이라 판단했습니다. 데이터는 **PostgreSQL 16**에 저장하고, 세션·캐시는 **Redis 7**로 분리했으며, 스키마 변경은 **Alembic 마이그레이션**으로 관리합니다.

ML 엔진은 K-Means·Isolation Forest·LSTM·XGBoost 네 모델을 별도 학습 파이프라인으로 두어, 서버 기동 시 자동으로 재학습되도록 구성했습니다.

여기서 중요한 설계 결정이 **마이크로서비스 분리**입니다. 오른쪽을 보시면, **fraud-service를 포트 8010에 독립 배포**했습니다. 사기 탐지는 모델 업데이트 빈도와 확장 요구가 다르기 때문에, 메인 서비스와 분리해 **각각 독립적으로 스케일아웃**할 수 있도록 했습니다. 두 서비스는 Docker Compose로 통합 관리되며, PostgreSQL·Redis까지 한 번에 기동됩니다.

## 🎯 슬라이드 5 — AI 소비 패턴 분석 `[3:20 ~ 4:50]` ⏱ **1분 30초**

핵심 AI 모델 네 가지입니다.

**첫째, K-Means 군집화**입니다. 식비·쇼핑·교통 등 **11개 카테고리별 지출 비율**을 표준화해 **4개 클러스터**로 분류합니다. 이를 통해 사용자를 절약형·균형형·소비형·투자형으로 자동 분류하고, 유형별로 예산 전략을 차별화합니다.

**둘째, Isolation Forest**입니다. 비지도 학습 모델로 **contamination 파라미터**를 조정하면서 이상 스코어를 실시간 산출합니다. 지도 학습과 달리 사기 라벨이 부족한 실제 환경에서도 동작한다는 점이 강점입니다.

**셋째, LSTM**입니다. 시계열 소비 데이터를 학습해 다음 달 카테고리별 지출액을 예측합니다. 단순 평균이 아니라 **시점 간 패턴**을 학습하기 때문에 급여일·공휴일 효과까지 반영됩니다.

**넷째, XGBoost**입니다. 과소비 레이블 기반 지도 학습 분류기로, **feature importance를 제공**하기 때문에 "왜 과소비로 판정되었는지"를 사용자에게 설명할 수 있습니다. 이것이 뒤에 나올 XAI 기능의 기반이 됩니다.

하단 파이프라인처럼 **데이터 수집 → 특성 추출 → ML 추론 → API 응답** 네 단계가 순차 실행되며, 분석 결과는 프론트엔드의 레이더·바·게이지 차트로 시각화됩니다.

## 🎯 슬라이드 6 — 사기 탐지 서비스 `[4:50 ~ 6:20]` ⏱ **1분 30초**

사기 탐지 서비스, fraud-service입니다.

왼쪽의 **판단 로직**부터 보시면, ML 스코어를 기준으로 네 단계로 분기합니다.
- **0.95 이상**이면 **BLOCK — 즉시 차단**,
- **0.35 이상**이면 **REVIEW — 추가 검토**,
- **0.005 이상**은 **SOFT_REVIEW — 경고 플래그**,
- 그 미만은 **PASS**입니다.

이 임계값은 Kaggle과 IEEE-CIS 데이터셋에서 **PR-AUC를 최대화하는 지점**으로 튜닝했습니다.

여기에 **규칙 엔진**을 병행합니다. 금액·시간·빈도·기기 변경 같은 다차원 규칙을 평가해서, **ML 판단과 규칙 판단 중 더 강한 쪽을 최종 액션으로 채택**합니다. REVIEW 이상이면 **Step-up Auth**가 자동 트리거되어 FIDO2 패스키 또는 TOTP 재인증을 요구합니다.

오른쪽 모델 구성을 보시면 세 모델을 구현했습니다.
- **Isolation Forest 비지도 모델** — Kaggle creditcard.csv 학습,
- **Random Forest 지도 모델** — open 데이터셋 기반,
- 그리고 **Hybrid IF+RF 모델** — IF의 이상 스코어를 피처로 주입해 RF가 최종 분류합니다.

세 모델은 **Stratified 5-fold CV**로 비교해 평균 PR-AUC가 가장 높은 모델을 권장 모델로 자동 선정하도록 했습니다.

## 🎯 슬라이드 7 — 부가 기능 `[6:20 ~ 7:40]` ⏱ **1분 20초**

부가 기능 여섯 가지입니다.

**XAI**는 XGBoost의 feature importance와 z-score를 곱해 **상위 3개 피처를 Reason Code로 반환**합니다. "심야 지출 비중이 평소 대비 3배 높음" 같은 자연어 설명을 사용자에게 제공합니다.

**FIDO 인증**은 단순 패스워드 방식의 한계를 보완하기 위해 도입했습니다. **TOTP(OTP 앱)와 FIDO2 패스키** 두 방식을 병행 지원하며, REVIEW 판정 시 Step-up Auth로 자동 트리거됩니다.

**감정 기반 분석**은 소비 직후 사용자가 입력한 감정 태그와 지출 내역을 상관분석해 **충동 소비 패턴을 탐지**합니다.

**예산 전략 추천**은 소비 유형별 맞춤 예산을 설계하고 카테고리 한도와 절약 팁을 제안합니다. 예산 80% 도달 시 실시간 Push 알림이 발송됩니다.

**금융 교육 콘텐츠**는 분석 결과를 기반으로 맞춤형 투자·저축 가이드를 제공합니다.

마지막으로 **보안 대시보드**입니다. 블록체인 기반 감사 로그로 모든 FDS 판정을 기록하고, MyData 동의 이력과 PID 가명처리 상태를 시각화합니다. 이 부분은 지도교수님의 요구사항을 반영한 결과물입니다.

## 💡 예상 질문 대비 답변

**Q1. 왜 Isolation Forest를 선택했나요?**
> 실제 사기 거래는 전체의 0.17% 수준으로 극단적 불균형입니다. 지도 학습은 라벨 수집에 한계가 있어, **라벨 없이 "평소와 다른 패턴"을 탐지하는 비지도 학습**이 1차 스크리닝에 적합합니다. 이후 RF와 결합한 Hybrid 모델로 정확도를 보강합니다.

**Q2. fraud-service를 왜 분리했나요?**
> 사기 탐지 모델은 **업데이트 주기와 트래픽 특성이 메인 서비스와 다릅니다.** 독립 배포해야 모델 교체·스케일아웃·장애 격리가 가능합니다. HTTP REST로 통신하며 환경변수로 URL을 주입해 배포 유연성을 확보했습니다.

**Q3. 블록체인 감사 로그를 왜 도입했나요?**
> FDS 판정은 금융 규제상 **감사 추적(Audit Trail)이 필수**입니다. 단순 DB 로그는 변조 가능하지만, Merkle root 기반 구조는 **사후 변조 탐지**가 가능합니다. 현재는 시뮬레이션 단계이며 운영급은 향후 작업입니다.

**Q4. 실제 사용자 데이터는 어떻게 확보하나요?**
> 현재는 Kaggle·IEEE-CIS 공개 데이터셋과 데모 시드 데이터로 학습하고 있습니다. 향후 계획의 **MyData 테스트베드 실연동**이 이 부분을 해결합니다.

**Q5. 개인정보 보호는 어떻게 하나요?**
> PID 가명처리로 원본 user_id를 해시화하고, 속성 기반 접근 제어로 열람 권한을 제한합니다. Federated Learning 시뮬레이션으로 **원본 데이터를 공유하지 않는 협력 학습 구조**까지 설계해 두었습니다.
